## Demo VM Bootstrap (user_data/demo_vm.sh)

This script runs once on first boot. It must be idempotent.
Sequence:
  1. apt-get: docker.io, docker-compose, certbot (or use Caddy for auto-TLS)
  2. Configure TLS: use Caddy with Cloudflare DNS challenge for demovm.ricardo.engineer
     - Caddy reverse-proxies HTTPS → web server on localhost:8080
     - PostgreSQL is NOT behind Caddy — it uses its own port 5432 with no TLS (private SG only)
  3. docker-compose up -d postgres web-server

docker-compose services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_DB: demo_app
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: (from SSM Parameter Store or Terraform output — never hardcoded)
    ports: ["5432:5432"]   # bound to 0.0.0.0 — SG restricts access
    volumes: [pgdata:/var/lib/postgresql/data]

  web-server:
    build: /opt/demo/web/
    environment:
      DB_HOST: localhost
      DB_NAME: demo_app
      DB_USER: postgres
      DB_PASS: (same as above — admin read access)
    ports: ["8080:8080"]
    restart: always

Schema (applied at bootstrap):
  CREATE TABLE IF NOT EXISTS secret_log (
    id          SERIAL PRIMARY KEY,
    username    TEXT NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    ttl_seconds INTEGER NOT NULL DEFAULT 300
  );

Add a log function that persists logs in a directory for monitoring:

```bash
LOGFILE="/var/log/bootstrap.log"

function log {
  local level="$1"
  local message="$2"
  local timestamp=$(date +"%Y-%m-%d %H:%M:%S")
  local log_entry="$timestamp [$level] - $message"

  echo "$log_entry" | sudo tee -a "$LOGFILE"
}

# This runs when ANY command fails
function on_error {
  local exit_code=$?
  local line_no=$1
  log "ERROR" "Script failed with exit code $exit_code at line $line_no"
}

trap 'on_error $LINENO' ERR
```

## Vault Config (infra/vault/) — HCP Vault Dedicated

vault_addr:       https://<HCP_VAULT_URL>   # already running, never provision
vault_namespace:  admin
provider:         hashicorp/vault

### Resources to create (all via vault provider):

1. vault_jwt_auth_backend "azure_ad"
   - path = "jwt"
   - oidc_discovery_url = "https://login.microsoftonline.com/<TENANT_ID>/v2.0"
   - bound_issuer = "https://login.microsoftonline.com/<TENANT_ID>/v2.0"

2. vault_jwt_auth_backend_role "github_actions"
   - backend = vault_jwt_auth_backend.azure_ad.path
   - role_name = "github-actions-demo"
   - role_type = "jwt"
   - bound_audiences = ["api://<AZURE_APP_CLIENT_ID>"]
   - bound_claims = { repository = "hashicardo/vault-pipeline-demo", environment = "demo" }
   - user_claim = "sub"
   - token_policies = ["demo-db-policy"]
   - token_ttl = 300   # 5 minutes — matches DB cred TTL

3. vault_database_secret_backend_connection "postgres"
   - backend = "database"
   - plugin_name = "postgresql-database-plugin"
   - connection_url = "postgresql://{{username}}:{{password}}@demovm.ricardo.engineer:5432/demo_app"
   - username = postgres root user (Terraform variable)
   - password = postgres root pass (Terraform variable)
   - allowed_roles = ["demo-role"]

4. vault_database_secret_backend_role "demo_role"
   - backend = "database"
   - name = "demo-role"
   - db_name = vault_database_secret_backend_connection.postgres.name
   - default_ttl = 300
   - max_ttl = 600
   - creation_statements:
       CREATE ROLE "{{name}}" WITH LOGIN PASSWORD '{{password}}'
       VALID UNTIL '{{expiration}}';
       GRANT SELECT, INSERT ON secret_log TO "{{name}}";
   - revocation_statements:
       REVOKE ALL ON secret_log FROM "{{name}}";
       DROP ROLE IF EXISTS "{{name}}";

5. vault_policy "demo_db"
   - name = "demo-db-policy"
   - policy:
       path "database/creds/demo-role" { capabilities = ["read"] }
       path "secret/data/demo/*"       { capabilities = ["read"] }

## GitHub Actions Pipeline (.github/workflows/demo-pipeline.yml)

runs-on: [self-hosted, demo]   # targets the runner VM

Environment: demo (required for bound_claims to match)

Jobs in order:

1. auth
   - uses: azure/login@v2 with client-id, tenant-id (OIDC, no secret)
   - runs: az account get-access-token --resource api://<CLIENT_ID>
   - exports JWT as VAULT_JWT (masked)
   - calls: vault write auth/jwt/login role=github-actions-demo jwt=$VAULT_JWT
   - exports VAULT_TOKEN (masked)

2. fetch-secret
   - vault read -format=json database/creds/demo-role
   - exports: DB_USER, DB_PASS (both masked)
   - exports: SECRET_CREATED_AT = $(date -u +%Y-%m-%dT%H:%M:%SZ)

3. use-secret (the wow moment)
   - psql "host=demovm.ricardo.engineer port=5432 dbname=demo_app
           user=$DB_USER password=$DB_PASS sslmode=disable"
   - INSERT INTO secret_log (username, created_at, ttl_seconds)
     VALUES ('$DB_USER', '$SECRET_CREATED_AT', 300);
   - prints: "Connected as $DB_USER — secret written to database"

4. show-expiry  (separate workflow, triggered manually after 5+ minutes)
   - attempts psql with same DB_USER / DB_PASS (stored as masked job outputs)
   - MUST FAIL with: FATAL: role "vault-token-xxxx" does not exist
   - pipeline step uses: continue-on-error: false
   - this failure IS the demo — do not fix it

### GitHub Actions secrets (set manually or via gh CLI):
  AZURE_CLIENT_ID, AZURE_TENANT_ID, AZURE_SUBSCRIPTION_ID, VAULT_ADDR

## Web Server — Secret Status UI (web/)

Language: Python (Flask) — simple, no build step, easy to read on screen during demo
Served at: https://demovm.ricardo.engineer (via Caddy reverse proxy → localhost:8080)

Behaviour:
  - Reads secret_log table using ADMIN credentials (postgres user) — hardcoded in env
  - Fetches the most recent row
  - Displays:
      "Secret username:   vault-token-xxxx"
      "Created at:        2025-04-07 14:32:01 UTC"
      "TTL:               300 seconds"
      "Expires at:        2025-04-07 14:37:01 UTC"
      [countdown timer in large text — JS, updates every second]
      If now() > expires_at: shows "DYNAMIC SECRET EXPIRED" in red, no countdown
  - Auto-refreshes the DB query every 10 seconds (in case a new pipeline run fires)
  - Page is intentionally minimal — this will be on a second screen during the demo

Dockerfile:
  FROM python:3.12-slim
  pip install flask psycopg2-binary
  COPY . /app
  CMD ["python", "app.py"]

The countdown and expiry logic runs entirely in browser JS using the
expires_at timestamp injected into the page by Flask. No websockets needed.

## Terraform Apply Order

Always apply in this sequence (dependencies flow downward):
  1. infra/azure/   → outputs tenant_id, client_id
  2. infra/aws/     → outputs demo_vm_public_ip (triggers Cloudflare DNS)
  3. infra/vault/   → needs DB reachable at demovm.ricardo.engineer:5432
                       needs azure outputs for JWT config

Destroy order: reverse (vault → aws → azure)

## Rules for Claude Code in this repo

- ALL infrastructure is Terraform. Never write raw AWS CLI commands or shell-based
  resource creation (except inside user_data bootstrap scripts).
- All Terraform must pass: terraform validate && terraform fmt -check
- All shell scripts (user_data) must start with: set -euo pipefail
- All scripts must be idempotent — safe to run more than once.
- Never hardcode IPs, passwords, or tokens in .tf files. Always use variables.
- Never provision HCP Vault — it already exists. Only configure it via vault provider.
- The expiry failure in the show-expiry job is intentional. Never fix it.
- The web server reads with admin DB credentials. Do not replace with dynamic creds.
- When scaffolding is complete, run: terraform validate in each module directory.
- The Cloudflare DNS A record for demovm.ricardo.engineer must be set to
  proxied = false (grey-cloud) — TLS is handled by Caddy on the VM, not Cloudflare.

## Demo Narrative (for README and customer-facing packaging)

1. "The pipeline authenticates using your existing Azure AD identity — no Vault token
    stored anywhere, no secret pre-shared with GitHub."
2. "Vault issues a unique database credential for this pipeline run only.
    It has never existed before and will never exist again."
3. "The pipeline writes to the database as that ephemeral user.
    The web UI shows you exactly who connected and when."
4. "Five minutes later — the countdown hits zero.
    The credential is gone. Not rotated. Not revoked. Just gone."
5. "Run the pipeline again — you get a completely different username."

Use this arc in the README. Do not invent alternative narratives.
